# Spec — Fase 2.3.8: Otimização de Assets + Refinamentos de UI

**Versão alvo:** v0.5.0  
**Data:** 2026-04-27  
**Fase anterior:** 2.3.7 (v0.4.0) — SVGs integrados, gargalo de performance identificado  
**Status:** spec aprovada, aguardando plano de implementação

---

## Contexto

O jogo está lento em devices Android medianos porque `flutter_svg` processa cada SVG em runtime. Com 26 assets (11 tiles + 11 hosts + 4 inventário), a decodificação é um gargalo perceptível. Esta fase resolve isso e faz 6 refinamentos de UI/UX que melhoram a experiência.

Os PNGs já existem nos mesmos diretórios com os mesmos nomes dos SVGs — só muda a extensão.

---

## Ordem de execução

```
A (SVG → PNG)          ← primeiro: desbloqueia performance, remove flutter_svg
         ↓
B, C, D, E             ← paralelo: independentes entre si após A concluído
         ↓
F, G                   ← último: cleanup de regras/dados
```

## A — Migrar todos os assets de SVG para PNG

### Objetivo
Remover `flutter_svg`, trocar todos os `SvgPicture.asset` por `Image.asset`, e implementar precache completo com `ResizeImage` dinâmico no boot.

### Arquivos a modificar
| Arquivo | Mudança |
|---|---|
| `pubspec.yaml` | Remover `flutter_svg`; manter declarações de diretórios (caminhos não mudam) |
| `lib/data/animals_data.dart` | Renomear `svgPath` → `tilePngPath`, `hostSvgPath` → `hostPngPath`; mudar extensões de `.svg` para `.png` em todos os 11 entries |
| `lib/data/models/animal.dart` | Renomear campos `svgPath` → `tilePngPath`, `hostSvgPath` → `hostPngPath` (ver seção 13.1 do design doc) |
| `lib/presentation/widgets/tile_widget.dart` | `SvgPicture.asset(animal.svgPath)` → `Image.asset(animal.tilePngPath, fit: BoxFit.contain)`; manter `Opacity(0.28)` |
| `lib/presentation/widgets/host_artwork.dart` | `SvgPicture.asset(animal.hostSvgPath)` → `Image.asset(animal.hostPngPath, fit: BoxFit.contain)`; tamanho do slot definido pelo item C |
| `lib/presentation/widgets/inventory_item_button.dart` | `SvgPicture.asset(svgPath)` → `Image.asset(pngPath, width: 32, height: 32)`; manter `color`+`colorBlendMode` pra estado disabled |
| `lib/app.dart` (widget raiz) | Chamar `preloadAnimalImages(context)` em `didChangeDependencies` do estado do widget raiz — garante `BuildContext` com `MediaQuery` disponível antes da `HomeScreen` |

### Arquivo a criar
| Arquivo | Conteúdo |
|---|---|
| `lib/core/utils/image_preloader.dart` | Função `preloadAnimalImages(BuildContext context)` — itera todos os 26 assets, calcula `renderSize` via `MediaQuery` (largura da tela ÷ 4 pra tiles × 1, × 2 pra hosts), chama `precacheImage(ResizeImage(AssetImage(path), width: renderSize, height: renderSize), context)` pra cada um |

### Assets a deletar
Todos os `.svg` em:
- `assets/images/animals/tile/` (11 arquivos)
- `assets/images/animals/host/` (11 arquivos)
- `assets/icons/inventory/` (4 arquivos)

### Estratégia de precache
- **Completo no boot:** todos os 26 PNGs pré-carregados antes da `HomeScreen` aparecer
- **`ResizeImage` dinâmico:** `renderSize` calculado uma vez via `MediaQuery.of(context).size.width`:
  - Tiles: `(screenWidth - padding) / 4` arredondado pro inteiro mais próximo
  - Hosts: `tileSize * 2` (slot 2×2)
  - Inventário: `32` (fixo — tamanho de render conhecido)

### Casos de teste obrigatórios
1. Cada um dos 11 níveis renderiza PNG correto no tile (snapshot test — regenerar goldens existentes)
2. Anfitrião renderiza PNG correto pra cada um dos 11 animais (snapshot test — regenerar goldens)
3. `InventoryItemButton` renderiza PNG correto pra `bomb_2`, `bomb_3`, `undo_1`, `undo_3`
4. Nenhum import de `flutter_svg` no projeto (lint/grep pós-migração)
5. Precache não causa lag visível na `HomeScreen` (teste manual em device Android médio)
6. `AnimalsGalleryScreen` exibe PNGs corretamente (atualizar pra usar `tilePngPath`/`hostPngPath`)

### Critérios de aceite
- App abre sem erro em debug e release
- Nenhuma referência a `flutter_svg` ou `SvgPicture` em qualquer arquivo `.dart`
- Tiles e anfitrião exibem os PNGs com qualidade visual equivalente aos SVGs
- FPS médio em 60s de gameplay visivelmente melhor em device Android médio vs v0.4.0

## B — Confirmação universal para todos os itens do inventário

### Objetivo
Padronizar o comportamento da `InventoryBar`: todo tap em item com contador > 0 abre `ConfirmUseDialog` antes de qualquer ação.

### Arquivos a criar
| Arquivo | Conteúdo |
|---|---|
| `lib/presentation/widgets/confirm_use_dialog.dart` | Widget de dialog reutilizável (ver spec abaixo) |

### Arquivos a modificar
| Arquivo | Mudança |
|---|---|
| `lib/presentation/widgets/inventory_item_button.dart` | `onTap` sempre abre `ConfirmUseDialog`; callback `onConfirm` executa a ação específica |

### Spec do `ConfirmUseDialog`
**Parâmetros:**
```dart
// Recebe:
String itemName         // "Bomba 2", "Desfazer 1", etc.
String effectDescription // "Explode 2 casas adjacentes escolhidas"
int currentCount        // contador atual do item
String pngPath          // caminho do PNG do ícone
VoidCallback onConfirm  // chamado ao tocar "Usar"
```

**Layout (de cima pra baixo):**
1. Ícone PNG grande do item (64×64dp, centralizado)
2. Título: "Usar [itemName]?" — Fredoka Bold, ~18sp
3. Sub-texto: `effectDescription` — Nunito Regular, ~14sp, cor secundária
4. Contador: "Você tem [currentCount] deste item" — Nunito SemiBold, ~13sp
5. Botões: `[Cancelar]` (outline) e `[Usar]` (filled, destaque)

**Animação de entrada:** fade + slide-up, 200ms  
**Som "Usar":** clique grave decisivo  
**Som "Cancelar":** clique neutro

### Fluxo unificado
```
Tap no item (contador > 0)
  → ConfirmUseDialog
      Cancelar → fecha, nada muda
      Usar →
        Desfazer 1/3: gameNotifier.undo(steps) → animação reversa 300ms → decrementa contador
        Bomba 2/3:    abre BombSelectionOverlay → jogador escolhe tiles → "Explodir" → animação 500ms → decrementa
```

**Tap em item com contador = 0:** nenhuma ação (botão acinzentado/desabilitado — comportamento já existente).

### Casos de teste obrigatórios
1. Tap Desfazer 1 → abre dialog; Cancelar → nada muda; Usar → desfaz + decrementa
2. Tap Desfazer 3 → mesma coisa
3. Tap Bomba 2 → abre dialog; Cancelar → nada muda; Usar → abre `BombSelectionOverlay`
4. Tap Bomba 3 → mesma coisa
5. Tap em item com contador 0 → nenhum dialog aberto
6. Snapshot test do `ConfirmUseDialog` pra cada um dos 4 itens

### Critérios de aceite
- Nenhum item do inventário executa ação sem confirmação prévia
- Cancelar nunca altera estado do jogo ou do inventário
- Dialog usa as convenções visuais do tema (botões ≥48dp, sombra, paleta)

## E — LivesIndicator redesenhado

### Objetivo
Substituir a fileira de corações por um indicador compacto e escalável: 1 coração com número sobreposto + badge contextual.

### Arquivos a modificar
| Arquivo | Mudança |
|---|---|
| `lib/presentation/widgets/lives_indicator.dart` | Refatorar completamente (ver spec abaixo) |
| `lib/presentation/screens/game/game_screen.dart` | Posicionar na linha do topo, lado esquerdo; tap pausa o jogo antes de abrir dialog |
| `lib/presentation/screens/home/home_screen.dart` | Posicionar no topo central |

### Spec do `LivesIndicator`

**Layout:**
```dart
Row(
  children: [
    Stack(  // coração com número
      alignment: Alignment.center,
      children: [
        Icon(Icons.favorite, size: 36, color: AppColors.heartRed),
        OutlinedText(text: '$current', style: Fredoka Bold 16sp, color: white),
      ],
    ),
    SizedBox(width: 6),
    _Badge(current: current),  // pill contextual
  ],
)
```

**Badge contextual:**
| Condição | Visual |
|---|---|
| `current < regenCap (5)` | Pill cinza, texto timer MM:SS regressivo |
| `current == regenCap (5)` | Pill verde `#66BB6A`, texto "Completo" |
| `current > regenCap (5)` | Pill amarelo `#FFD54F`, texto "Bônus ⭐" |

**Timer:** implementado com `SingleTickerProviderStateMixin` — `Ticker` atualiza a cada segundo calculando `nextRegenAt.difference(DateTime.now())` e formatando como MM:SS. Pausa automaticamente quando o widget sai de tela.

**Tap na GameScreen:** chama `gameNotifier.pause()` antes de abrir o dialog explicativo. Dialog fecha → `gameNotifier.resume()`.

**Tap na HomeScreen:** abre dialog explicativo direto (sem pausar nada).

**Animações:**
- Vida ganha: `flutter_animate` scale 1→1.15→1, 300ms no coração; badge atualiza
- Vida perdida: `flutter_animate` rotate ±5°, 200ms; número decrementa

### Casos de teste obrigatórios
1. `current=1` → coração "1" + pill cinza com timer
2. `current=4` → coração "4" + pill cinza com timer
3. `current=5` → coração "5" + pill verde "Completo"
4. `current=10` → coração "10" + pill amarelo "Bônus ⭐"
5. `current=99` → coração "99" + pill amarelo "Bônus ⭐" (sem overflow)
6. Timer atualiza a cada segundo (mock `DateTime`)
7. Tap na `GameScreen` pausa o jogo antes de abrir dialog
8. Snapshot test pra estados: 1, 4, 5, 10, 99

### Critérios de aceite
- Indicador nunca estoura o layout independente do valor de `current`
- Timer para quando o widget sai de tela (sem leak de `Ticker`)
- Jogo sempre pausado antes de abrir dialog na `GameScreen`

## F — Sistema de vidas: regras de armazenamento no código

### Objetivo
Garantir que o código reflete as regras da seção 5 do design doc: regen cap = 5, earned cap = 15, purchased sem cap.

### Migração de dados
**Estratégia:** reset completo do `LivesState` no Hive — fase de desenvolvimento, sem usuários reais.

**Implementação:** no boot, `LivesRepository` verifica se o schema é v0.4.0 (ausência de `regenCap`/`earnedCap`). Se sim, apaga e reinicializa com defaults: `current=5, regenCap=5, earnedCap=15, nextRegenAt=null, adWatchesToday=0`.

### Arquivos a modificar
| Arquivo | Mudança |
|---|---|
| `lib/data/models/lives_state.dart` | Adicionar `regenCap` (const 5) e `earnedCap` (const 15) |
| `lib/domain/lives_system/lives_notifier.dart` | Implementar `regenerate()`, `addEarned(n)`, `addPurchased(n)`, `consume()` conforme spec abaixo |
| `lib/data/repositories/lives_repository.dart` | Adicionar `clearAndReinit()` + verificação de schema no boot |

### Spec dos métodos do `LivesNotifier`
```
regenerate()
  - guard: se current >= regenCap → no-op
  - current += 1
  - se current < regenCap → nextRegenAt = now + 30min
  - se current >= regenCap → nextRegenAt = null

addEarned(int amount)
  - current = min(current + amount, earnedCap)
  - se current >= regenCap → nextRegenAt = null

addPurchased(int amount)
  - current = current + amount  // sem cap
  - se current >= regenCap → nextRegenAt = null

consume()
  - guard: se current <= 0 → no-op (game over sem vidas não deve acontecer)
  - current -= 1
  - se current < regenCap e nextRegenAt == null → nextRegenAt = now + 30min
```

### Casos de teste obrigatórios
1. Estado inicial após reinit: `current=5, regenCap=5, earnedCap=15, nextRegenAt=null`
2. `consume()` com `current=5` → `current=4`, `nextRegenAt` agendado pra +30min
3. `consume()` com `current=6` → `current=5`, `nextRegenAt=null` (ainda >= regenCap)
4. `regenerate()` com `current=4` → `current=5`, `nextRegenAt=null`
5. `regenerate()` com `current=5` → no-op
6. `addEarned(5)` com `current=14` → `current=15` (clamp em earnedCap)
7. `addEarned(5)` com `current=15` → `current=15` (clamp)
8. `addPurchased(10)` com `current=14` → `current=24`
9. `addPurchased(10)` com `current=50` → `current=60`
10. Boot com schema antigo → `clearAndReinit()` chamado → estado inicial correto

### Critérios de aceite
- Nenhum caminho de código permite `current` negativo
- `addEarned` nunca ultrapassa `earnedCap`
- `addPurchased` nunca é bloqueado por cap
- `nextRegenAt` é `null` sempre que `current >= regenCap`
