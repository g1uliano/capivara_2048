## Correção A — Vida só consome no Game Over

### Problema
Vida é consumida ao chamar `startNewGame()`. Se o jogador sai antes do game over, perde vida injustamente.

### Decisões
- `consumeLife()` removido de `startNewGame()` / notifier de partida
- `consumeLife()` chamado apenas quando o game engine detecta `isGameOver == true` (tabuleiro travado — sem movimentos possíveis). Não é disparado pelo botão "novo jogo" nem pelo botão de voltar ao menu.
- Vitória (tile 2048 formado) **não** consome vida
- Pré-condição em `startNewGame()`: se `livesState.current == 0`, redirecionar para `NoLivesScreen` sem iniciar partida

### Migração (primeira abertura pós-update)
- Flag `lives_reset_v235` (bool) no Hive, default `false`
- Se flag ausente ou `false`: resetar `livesState.current = 5`, setar flag `true`
- Vidas anteriores descartadas; jogador recomeça com 5
- Sem notificação explícita ao jogador (mudança silenciosa)

### Arquivos afetados
- `lib/domain/lives/` — lógica de consumo
- `lib/presentation/controllers/` — notifier que chama `startNewGame`
- `lib/data/repositories/` — migração da flag no Hive

### Casos de teste obrigatórios
| Cenário | Resultado esperado |
|---|---|
| Sair pro menu durante partida | vidas inalteradas |
| Reiniciar partida em andamento | vidas inalteradas |
| Game over | vidas −1 |
| Vitória (2048 formado) | vidas inalteradas |
| `current == 0` ao tentar iniciar | bloqueado, exibe `NoLivesScreen` |
| Primeira abertura pós-update | `current == 5`, flag `lives_reset_v235 = true` |

### Critério de aceite
Todos os 6 casos de teste passam. Flag de migração setada corretamente no Hive.

## Correção E — Mapa explícito de cores de fundo

### Problema
`Color.lerp(borderColor, menta, 0.65)` no `GameBackground` mistura rosa do Boto com verde-menta e gera bege/cinza sem identidade. Outros animais também podem ter cores dessaturadas.

### Decisões
- Novo campo `backgroundBaseColor` (`Color`) no modelo `Animal`
- `animals_data.dart` populado com 11 cores explícitas (tabela abaixo)
- Lógica `Color.lerp` automática removida do `GameBackground`
- `GameBackground` lê `animal.backgroundBaseColor` diretamente

| Animal | `borderColor` (tile) | `backgroundBaseColor` (fundo) |
|---|---|---|
| Tanajura | `#C0392B` | `#F5C2BA` |
| Lobo-guará | `#E67E22` | `#FAD3B2` |
| Sapo-cururu | `#8D6E63` | `#D7C4BC` |
| Tucano | `#FFB300` | `#FFE9A8` |
| Arara-azul | `#1E88E5` | `#B5D7F4` |
| Preguiça | `#BCAAA4` | `#E8E0DC` |
| Mico-leão-dourado | `#FF8F00` | `#FFD7A1` |
| Boto-cor-de-rosa | `#F48FB1` | `#FBD0DD` |
| Onça-pintada | `#FBC02D` | `#FFEFB0` |
| Sucuri | `#2E7D32` | `#BFD9C0` |
| Capivara Lendária | `#FFD54F` | `#FFEFB8` |

### Arquivos afetados

- `lib/presentation/widgets/game_background.dart` — remover lerp, ler do modelo

| Tucano renderizado | fundo amarelo pastel (`#FFE9A8`) |

### Critério de aceite
Cada animal renderiza exatamente a `backgroundBaseColor` definida no modelo. Lógica de `lerp` automática ausente do codebase.

## Correção B — Transição de fundo sem flicker

### Problema
Ao trocar de animal anfitrião, a tela pisca. Causa: rebuild da árvore inteira ao invés de interpolação isolada da camada de cor.

### Decisões
- `TweenAnimationBuilder<Color?>` aplicado **apenas à camada de cor de fundo**, isolada por `RepaintBoundary`
- Textura geométrica (camada acima da cor): `AnimatedSwitcher` com `CrossFadeState`, `ValueKey(animal.id)` — não recria a cada frame
- Estado da cor anterior fica **interno ao `GameBackground`** — sem `setState` no widget pai durante a transição
- Duração: 700ms, curve `easeInOut`

### Arquivos afetados
- `lib/presentation/widgets/game_background.dart` — refactor da estratégia de animação

### Casos de teste obrigatórios
| Cenário | Resultado esperado |
|---|---|
| Transição Tucano→Boto | 3 frames intermediários mostram interpolação gradual (snapshot) |
| Durante transição | `BoardWidget` não rebuilda (verificar com `RepaintBoundary` + debug paint) |
| Performance no emulador | FPS ≥ 50 durante transição |

### Critério de aceite
Sem salto abrupto de cor. `BoardWidget` protegido por `RepaintBoundary`. FPS ≥ 50.

## Correção D — OutlinedText para legibilidade

### Problema
Texto branco sobre fundos amarelos/dourados (Tucano, Onça, Capivara) fica difícil de ler.

### Decisões
**Dois mecanismos complementares:**

**Helper `outlinedWhiteTextStyle(TextStyle base)`** — casos simples:
- Função em `lib/core/theme/text_styles.dart`
- Aplica 4 `Shadow` pretas com offset `(±1.5, ±1.5)` ao `TextStyle` recebido
- Drop-in em qualquer `Text` existente sem alterar a árvore de widgets

**Widget `OutlinedText`** — casos com controle fino (texto animado, etc.):
- `lib/presentation/widgets/outlined_text.dart`
- Stack: `Text` fill branco + `Text` stroke preto 1.5px via `Paint`

**Onde aplicar:**
- Nome do anfitrião (`HostBanner` / `HostArtwork`)
- Cronômetro, pontuação, recorde (`StatusPanel`)
- Todo `Text` branco sobre `GameBackground`

**Onde NÃO aplicar:**
- Textos sobre tiles (fundo branco/pastel sólido)

### Arquivos afetados
- `lib/core/theme/text_styles.dart` — helper `outlinedWhiteTextStyle`
- `lib/presentation/widgets/outlined_text.dart` — novo widget
- `lib/presentation/widgets/host_banner.dart` (ou `host_artwork.dart`) — aplicar estilo
- `lib/presentation/widgets/status_panel.dart` — aplicar estilo

### Casos de teste obrigatórios
| Cenário | Resultado esperado |
|---|---|
| Cronômetro branco sobre fundo Tucano (amarelo) | contorno preto visível (snapshot) |
| Nome do animal sobre fundo Capivara (dourado) | legível (snapshot) |
| Contraste WCAG | texto + contorno alcança nível AA em todas as 11 cores |

### Critério de aceite
Snapshots passam. Contraste AA verificado nas 11 cores.

## Correção C — Pause sem sobreposição via LayoutBuilder

### Problema
Em algumas telas (especialmente com cronômetro HH:MM:SS), o botão de pause flutuante fica sobre o recorde no `StatusPanel`.

### Decisões
- Manter botão de pause flutuante (sem redesign do `StatusPanel`)
- `LayoutBuilder` (ou `Builder` + `context.findRenderObject`) calcula a posição inferior do `StatusPanel` em tempo de layout
- Botão posicionado com margem mínima de 12dp abaixo do panel
- Área de toque mínima: 48×48dp

### Arquivos afetados
- `lib/presentation/screens/game/` — tela do jogo onde o pause é posicionado

### Casos de teste obrigatórios
| Cenário | Resultado esperado |
|---|---|
| Tela 360×640 | pause não toca `StatusPanel` |
| Tela 412×915 | pause não toca `StatusPanel` |
| Cronômetro HH:MM:SS (panel em altura máxima) | pause mantém 12dp de gap |
| Área de toque | ≥ 48×48dp em todos os casos |

### Critério de aceite
4 casos de teste passam. Sem sobreposição em nenhuma resolução testada.
